## Reading path

The reader walks the page in the order it is written. Each step below maps a
reader goal to the heading in `index.en.mdx` that carries it, and records a
one-line reader-perspective confirmation.

### Step 0 — Frontmatter and opening framing
- Landing section: frontmatter + opening three paragraphs (lines 1–14).
- Reader goal: know what this page is for and what they will have at the end.
- Reader-perspective confirmation: title reads `Object Storage`, description
  names the concrete outcome ("create a bucket and upload your first file
  through the console"), the opening sentence declares this is the start-here
  guide, and the third paragraph promises a concrete end-state (a working
  bucket with one file in it plus the four credential values). The reader
  knows what they are about to do and what success looks like at a high
  level before any step begins.

### Step 1 — Open the Object Storage app
- Landing section: `## What Object Storage Is` (product framing) + first step
  of `## Create a Bucket` → `<h4>Open the Object Storage app</h4>`.
- Reader goal: understand the product, then open the app.
- Reader-perspective confirmation: the product framing is two short
  paragraphs that explain Object Storage is S3-compatible, built on MinIO,
  and managed by Sealos — enough to understand what they are using without
  marketing filler. The first numbered step under Create a Bucket tells them
  to sign in to the Sealos console and open the Object Storage app from the
  application launcher, and states the app lists existing buckets and
  exposes the entry point for creating a new one. The reader can open the
  app from this step alone.

### Step 2 — Create a bucket with `private` permission
- Landing section: `## Create a Bucket` → `<h4>Create a new bucket</h4>`
  step + the `private` / `publicRead` / `publicReadWrite` Callout.
- Reader goal: create a bucket and choose the correct permission.
- Reader-perspective confirmation: the step tells them to use the bucket
  creation action, enter a name, and choose `private` as the access
  permission. The Callout explains all three permission levels in one
  place, explicitly recommends `private` for the first bucket, and notes
  `publicReadWrite` should be avoided in production. The reader knows to
  pick `private`, knows what the other two options are, and knows the new
  bucket appears in the bucket list after they apply the change. No
  external guide is needed.

### Step 3 — Get credentials
- Landing section: `## Get Your Credentials` section body + the Internal /
  External endpoint Callout.
- Reader goal: know what credential values exist and where to read them.
- Reader-perspective confirmation: the section names the four values
  explicitly (Access Key, Secret Key, Internal endpoint, External
  endpoint), tells the reader to select the bucket in the bucket list and
  use the access key action to open the credential display, and states the
  values are read directly from the popup. The Callout makes the Internal
  vs External split clear with one-line operational framing ("for services
  running inside Sealos" vs "for access from outside"). The reader
  understands what they will see, why there are two endpoints, and which
  one applies to them — without needing an SDK setup guide.

### Step 4 — Upload a file
- Landing section: `## Upload Your First File` → three `<h4>` sub-steps
  (`Open the bucket`, `Upload a file`, `Confirm the upload`).
- Reader goal: pick a test file and complete a console upload.
- Reader-perspective confirmation: the sub-steps walk the reader from
  opening the bucket, to using the upload action in the bucket toolbar, to
  picking a small test file (plain text or small image under 1 MB is
  explicitly recommended), to waiting for the upload to complete. The
  reader knows what file to use, where to click, and what "done" feels
  like mid-flow, without guessing.

### Step 5 — Verify first success in the bucket's file list
- Landing section: `## Verify Your Upload`.
- Reader goal: confirm the file is really there in a way consistent with D-06.
- Reader-perspective confirmation: the section defines first success as
  two signals checked in order — (1) a file row in the bucket's file list
  showing expected name, size, and upload timestamp, and (2) an object
  detail view showing an addressable object path / URL. It explicitly
  notes the bucket is private and the URL is access-controlled, and
  states the guide does not ask the reader to make the bucket public or
  generate a signed URL. The reader knows precisely what "first upload
  confirmed" looks like and that they are not expected to go further in
  this guide.

### Step 6 — If the first upload fails
- Landing section: `## If the first upload fails`.
- Reader goal: have clear first-check guidance if Step 4 or Step 5 does
  not produce the expected result.
- Reader-perspective confirmation: the section is a five-item ordered
  list with bucket-first / credential-side / file-side grouping, each
  item carrying one short operational hint. It does not send the reader
  on a decision tree and does not widen into an SDK tutorial. The reader
  has a single compact recovery path to try first, in the correct order,
  without flipping to another page.

### Step 7 — Scope boundary
- Landing section: `## What's Next`.
- Reader goal: know what the page is closing and what it is not promising.
- Reader-perspective confirmation: the section affirms the end-state
  ("you now have a working bucket with your first file stored in it and
  the four credential values ready"), then names further capabilities
  (MinIO-compatible SDKs, static website hosting, built-in monitoring)
  as things to explore later — explicitly not walkthroughs the reader
  needs to follow now. This preserves the milestone's single start-here
  posture and does not over-promise. The final sentence points Chinese
  readers to the Chinese version of the page, which is in sync.

## First-failure recovery

`## If the first upload fails` opens with "Check these first, in order:" and
enumerates five fixed checks in the locked D-09 order: (1) wrong or missing
bucket, (2) permission denied, (3) wrong endpoint, (4) wrong credentials,
(5) upload size or file-name issue. Each item carries one short
operational hint — checks 1, 2, and 5 say "Look here first" and target the
console UI context; checks 3 and 4 explicitly frame themselves as
"Matters when you move to SDK or CLI access", which is exactly the D-11
framing. The five-item checklist therefore covers the bucket / permission /
endpoint / credentials / size-or-filename axes in the fixed bucket-first,
credential-side, file-side order required by D-09, without decision trees,
retry advice, or network troubleshooting filler.

## Clarity / completeness concerns (non-blocking, recorded only)

These are observations recorded for the phase verifier to weigh. None are
treated as blockers in this wave and none are fixed here — any follow-up
belongs in a gap-closure plan after the verifier call.

1. **Permission Callout ordering.** The `private` / `publicRead` /
   `publicReadWrite` Callout explains the three permission levels after the
   step that already told the reader to pick `private`. A reader who wants
   to understand why they are picking `private` before picking it has to
   jump ahead to the Callout and back. This is a reading-flow nit, not a
   correctness problem — the reader is still told the right answer at the
   right moment.
2. **"Move to SDK or CLI access" framing appears only in the failure
   checklist.** Checks 3 and 4 of the first-failure section frame endpoint
   and credential issues as things that "matter when you move to SDK or
   CLI access." The Get Your Credentials section earlier in the page uses
   similar framing ("having them on hand prepares you for any future SDK
   or API access"), so the two are consistent, but a reader who has
   already copied an endpoint into a client of some kind might briefly
   wonder if those checks apply to them today. Not a blocker — the page
   is console-only by design (D-02) and the framing is accurate.
3. **Private-bucket URL framing in Verify Your Upload.** The verification
   section says the object detail view shows a URL that "is
   access-controlled" because the bucket is private, and explicitly tells
   the reader this guide does not ask them to generate a signed URL. A
   reader who clicks the URL to test it and gets denied could momentarily
   interpret that as a failure rather than the expected behavior. The
   text is technically correct and aligns with D-07. A future pass could
   pre-empt the "I clicked it and it didn't open" reaction with one more
   sentence; not required for publish-safety.
4. **Upload file size guidance is soft.** The Upload a file sub-step
   recommends "a small test file — a plain text file or a small image
   under 1 MB" and the failure checklist mentions "file too large for
   console upload" as check 5. The actual console upload limit is not
   named and does not need to be named (product-truth stability concern),
   but a reader who fails on a 50 MB file will land on check 5 and the
   hint "use a small test file" without a concrete threshold. Acceptable
   for a start-here guide; recorded for completeness.

None of these reach the "breaks the end-to-end reading path" threshold.
