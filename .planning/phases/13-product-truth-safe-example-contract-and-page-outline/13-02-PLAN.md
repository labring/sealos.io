**## Console-UI first-upload example contract**

Per D-01 and D-02, Phase 14 must use exactly one canonical first-upload posture: a console-UI walkthrough (no code, no SDK, no CLI).

### Required example flow (per D-02)
The example must follow this exact sequence:
1. Create a bucket (set permission to `private` per D-07)
2. Get credentials (click the "Access Key" button in the console per D-05)
3. Upload a file via the console Upload button (per D-01)
4. Verify success: the uploaded file appears in the console file list (per D-08)

### Credential display rules (per D-05)
- Direct users to click the "Access Key" button in the console
- The credential display shows: Access Key, Secret Key, Internal endpoint, External endpoint
- The page must describe this as a console-guided retrieval action
- The page must NOT display placeholder credential values (no `xxxxxxxx` patterns, no `your-access-key-here`)
- This approach differs from AI Proxy v1.2 (which used environment variables) because Object Storage v1.3 is console-UI-first per D-06

### File type and size recommendation (Claude's Discretion)
- Recommend uploading a small test file (e.g., a text file or small image)
- Keep the recommendation generic enough that any user can follow it
- Do not require a specific file format or exact size limit

### What the example must NOT include (per D-03)
- No SDK code examples (Go, Java, Node.js, Python)
- No CLI upload commands
- No `minio-go` or `minio` npm import examples
- No programmatic credential handling

**## Permission guidance contract (per D-07, D-09)**

### Default recommendation
- The example bucket must use `private` permission (safest default)
- The page must recommend `private` as the first-bucket choice

### Permission level explanations
The page must briefly explain all three levels (per D-09):
- `private`: only authenticated users can access bucket contents (recommended for first bucket)
- `publicRead`: anyone can read objects without authentication; write requires credentials
- `publicReadWrite`: fully open read/write access (use with caution)

### Permission scope for v1.3
- The example stays within `private` permission throughout
- Public URL verification is out of scope for the first-upload path (per D-08)
- Permission editing guidance is reference-only, covered briefly

**## Verification signal contract (per D-08)**

### First-success definition
- First-success = the uploaded file appears in the console file list
- The page must describe what the user should see after upload: the file name, size, and timestamp visible in the bucket's file list view

### What verification must NOT include
- No public URL access test (bucket is private)
- No SDK-based verification
- No `curl` or `wget` download test

**## Product-truth constraints (per D-10)**

### Stable facts (may state as exact values)
- Object Storage is S3-compatible
- Credentials include Access Key, Secret Key, Internal endpoint, External endpoint
- Three permission levels exist: private, publicRead, publicReadWrite
- The product supports file upload, download, permission management
- MinIO-compatible SDK access is available (mention only, no examples in v1.3)

### Unstable facts (use neutral descriptions until Phase 14 live-verifies)
- Exact button labels in the console (e.g., the precise text on the Upload button, the Access Key button)
- Exact menu navigation paths
- Console URL patterns (e.g., `cloud.sealos.run` may change)
- Exact file size limits or storage quotas
- Exact UI layout or position of controls

### Handling rule
- Stable facts: write as direct statements with exact values
- Unstable facts: use generic action-oriented descriptions (e.g., "click the upload control" rather than "click the Upload button labeled 'Upload'")
- When in doubt, prefer neutral descriptions that remain correct even if the UI changes

**## Screenshot posture for Phase 14**

Reference the audit's screenshot classifications. For the English page:
- First-upload-path screenshots (bucket creation, file upload, Access Key entry, Access Key result): may be reused with English alt text per D-04, noting they show Chinese UI
- Deferred screenshots (static hosting, custom domain): omit from v1.3 English page
- Decorative screenshots (app launcher): omit, describe navigation in text instead

**## Deferred content boundary**

Explicitly list what Phase 14 must NOT include:
- SDK integration examples (deferred per D-03, ENHC-01)
- Static website hosting walkthrough (deferred, ENHC-02)
- Monitoring and resource metrics (deferred, ENHC-03)
- Custom domain configuration (deferred, ENHC-04)
- Multi-page Object Storage navigation
